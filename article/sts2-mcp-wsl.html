Connect STS2 MCP from WSL to the STS2 Game

This is the setup I used to let a Codex session running in WSL control Slay the Spire 2 running on Windows through the STS2_MCP mod. The tricky part is that the game mod listens on Windows loopback, while Codex and the Python MCP bridge run inside WSL.

Keep this local. The MCP endpoint can send actions to the game, so do not expose it through a public tunnel or a broad LAN listener. Bind the proxy to the WSL adapter IP, restrict the firewall rule to that address, and remove the rule when you are done.
The final working route was: 
Codex in WSL
  -> Python MCP bridge
  -> http://<WSL_ADAPTER_IP>:15527
  -> host portproxy bound to the WSL adapter
  -> http://127.0.0.1:15526
  -> STS2_MCP mod inside the game

One Snippet

If I only want the operational commands, this is the compact version. Run the Windows part in Administrator PowerShell, then run the WSL part from the Linux shell. 

# Windows PowerShell, as Administrator
curl.exe -s http://127.0.0.1:15526/
$sts2McpHost = Get-NetIPAddress -InterfaceAlias "vEthernet (WSL (Hyper-V firewall))" -AddressFamily IPv4 | Select-Object -ExpandProperty IPAddress
$sts2WslRemote = (wsl.exe hostname -I).Trim().Split()[0]
netsh interface portproxy delete v4tov4 listenaddress=$sts2McpHost listenport=15527
netsh interface portproxy add v4tov4 listenaddress=$sts2McpHost listenport=15527 connectaddress=127.0.0.1 connectport=15526
New-NetFirewallRule -DisplayName "STS2 MCP from WSL 15527" -Direction Inbound -Action Allow -Protocol TCP -LocalAddress $sts2McpHost -LocalPort 15527 -RemoteAddress $sts2WslRemote
$sts2McpHost

# WSL. Replace <WSL_ADAPTER_IP> with the value printed by $sts2McpHost.
export STS2_MCP_HOST=<WSL_ADAPTER_IP>
curl http://$STS2_MCP_HOST:15527/

# ~/.codex/config.toml
[mcp_servers.sts2]
command = "python3"
args = ["<STS2MCP_REPO>/mcp/server.py", "--base-url", "http://<WSL_ADAPTER_IP>:15527", "--no-trust-env"]
enabled = true
The adapter IP is not stable. In my run it was 172.29.224.1, but Windows or WSL can assign a different value after restart. Always use the current value printed by $sts2McpHost when testing from WSL and when editing ~/.codex/config.toml.
There is no auth token. Treat anything that can reach this HTTP API as having full game control. That includes Codex, scripts, agents, and dependencies running inside the same WSL VM.

What Was Running

  • Slay the Spire 2 ran on Windows.
  • The STS2_MCP mod exposed a local HTTP API on Windows.
  • Codex ran in WSL and used the repo at <STS2MCP_REPO>.
  • The Python MCP bridge was STS2MCP/mcp/server.py.

The mod itself was live on Windows: 

curl.exe -s http://127.0.0.1:15526/

Expected response:

{
  "message": "Hello from STS2 MCP v0.4.0",
  "status": "ok"
}

The Problem

The default MCP bridge used http://localhost:15526. Inside WSL, localhost means the Linux VM, not Windows. So the MCP bridge returned: 

Error: Cannot connect to STS2_MCP mod. Is the game running with the mod enabled?

Directly trying the WSL DNS proxy address, such as 10.255.255.254:15526, also failed in this environment. The stable answer was to create a host port proxy on a separate port and connect to the WSL adapter IP.

Windows Commands

1 Confirm the game mod is alive

Run this in Windows PowerShell:

curl.exe -s http://127.0.0.1:15526/

Use curl.exe, not curl, because PowerShell aliases curl to Invoke-WebRequest.

2 Create a WSL-facing port proxy

Run PowerShell as Administrator:

$sts2McpHost = Get-NetIPAddress -InterfaceAlias "vEthernet (WSL (Hyper-V firewall))" -AddressFamily IPv4 | Select-Object -ExpandProperty IPAddress
$sts2WslRemote = (wsl.exe hostname -I).Trim().Split()[0]
netsh interface portproxy delete v4tov4 listenaddress=$sts2McpHost listenport=15527
netsh interface portproxy add v4tov4 listenaddress=$sts2McpHost listenport=15527 connectaddress=127.0.0.1 connectport=15526
netsh interface portproxy show v4tov4

I used 15527 as the WSL-facing port. The game already owns 15526, so keeping the proxy on a different port avoids an ambiguous setup. Binding the proxy to the WSL adapter IP is safer than listening on 0.0.0.0, which can make the port reachable from interfaces that do not need game control access.

3 Allow the port through Windows Firewall

New-NetFirewallRule -DisplayName "STS2 MCP from WSL 15527" -Direction Inbound -Action Allow -Protocol TCP -LocalAddress $sts2McpHost -LocalPort 15527 -RemoteAddress $sts2WslRemote

This keeps the rule scoped to the WSL adapter address instead of opening the port on every network interface. It also scopes the source to the current WSL VM IP instead of LocalSubnet, which can include other local networks such as a home LAN.

Avoid broad Hyper-V firewall changes for the normal setup. A command like the one below changes the default inbound policy for a WSL VM and is wider than this STS2 bridge needs. Use it only as a temporary diagnostic on your own machine, then restore the previous policy. 
Set-NetFirewallHyperVVMSetting -Name "{40E0AC32-46A5-438A-A0B2-2B479E8F2E90}" -DefaultInboundAction Allow

Restore the default policy after the diagnostic:

Set-NetFirewallHyperVVMSetting -Name "{40E0AC32-46A5-438A-A0B2-2B479E8F2E90}" -DefaultInboundAction NotConfigured

4 Find the Windows WSL adapter IP

Get-NetIPAddress -InterfaceAlias "vEthernet (WSL (Hyper-V firewall))" -AddressFamily IPv4 | Select-Object -ExpandProperty IPAddress

In my run, this printed: 

172.29.224.1
Treat this as an example, not a permanent address. This IP can change after WSL or Windows restarts. If the bridge stops working later, run this command again and update every <WSL_ADAPTER_IP> value in the WSL test command and Codex MCP base URL.

5 Test the proxy from Windows

curl.exe -v http://${sts2McpHost}:15527/

This should return the same JSON as port 15526. If this fails, fix the host portproxy before debugging WSL.

WSL Commands

1 Test the Windows-facing URL from WSL

curl http://<WSL_ADAPTER_IP>:15527/

Expected response:

{
  "message": "Hello from STS2 MCP v0.4.0",
  "status": "ok"
}

2 If it fails, compare these checks

In Windows PowerShell:

netsh interface portproxy show v4tov4
netstat -ano | findstr :1552
Get-NetIPAddress -InterfaceAlias "vEthernet (WSL (Hyper-V firewall))" -AddressFamily IPv4 | Select-Object -ExpandProperty IPAddress

In WSL:

cat /etc/resolv.conf
curl -v --connect-timeout 3 http://<WSL_ADAPTER_IP>:15527/

The /etc/resolv.conf nameserver is not always the address you want. In this setup, 10.255.255.254 was reachable as a DNS proxy, but it refused the STS2 connection. The correct target was the vEthernet (WSL) IP from PowerShell.

Codex MCP Config

Codex reads MCP servers from ~/.codex/config.toml. The working server entry points the STS2 bridge to the WSL-reachable URL. Replace <WSL_ADAPTER_IP> with the current PowerShell value from $sts2McpHost: 

[mcp_servers.sts2]
command = "python3"
args = ["<STS2MCP_REPO>/mcp/server.py", "--base-url", "http://<WSL_ADAPTER_IP>:15527", "--no-trust-env"]
enabled = true

After changing this file, restart Codex so the MCP server process is relaunched with the new arguments.

Code Change I Made

The original bridge only accepted --host and --port, then built a URL like http://localhost:15526. I changed STS2MCP/mcp/server.py so it can accept a full base URL: 

python3 <STS2MCP_REPO>/mcp/server.py --base-url http://<WSL_ADAPTER_IP>:15527 --no-trust-env

The bridge also reads an environment variable: 

STS2_MCP_BASE_URL=http://<WSL_ADAPTER_IP>:15527 python3 <STS2MCP_REPO>/mcp/server.py --no-trust-env

Conceptually, the change was:

  • Add --base-url to the CLI.
  • Add STS2_MCP_BASE_URL as an environment override.
  • Use one shared request helper so all singleplayer, multiplayer, profile, compendium, wiki, and profile-list calls use the selected base URL.
  • Keep the default behavior as http://localhost:15526 for normal non-WSL setups.

The config edit is intentionally small: the MCP bridge still speaks stdio to Codex, and the game mod still speaks HTTP locally on Windows. Only the bridge-to-game URL changed.

Minimal Patch

If the upstream bridge does not have --base-url yet, this is the small shape of the local patch. It adds a full URL override without changing the default localhost:15526 behavior. 

diff --git a/mcp/server.py b/mcp/server.py
--- a/mcp/server.py
+++ b/mcp/server.py
@@
-import json
+import json
+import os
@@
-_base_url: str = "http://localhost:15526"
+_base_url: str = os.environ.get("STS2_MCP_BASE_URL", "http://localhost:15526").rstrip("/")
@@
 def main():
     parser = argparse.ArgumentParser(description="STS2 MCP Server")
+    parser.add_argument("--base-url", type=str, default=None, help="Full game HTTP server URL, e.g. http://127.0.0.1:15526")
     parser.add_argument("--port", type=int, default=15526, help="Game HTTP server port")
     parser.add_argument("--host", type=str, default="localhost", help="Game HTTP server host")
@@
-    global _base_url, _trust_env
-    _base_url = f"http://{args.host}:{args.port}"
+    global _base_url, _trust_env
+    if args.base_url:
+        _base_url = args.base_url.rstrip("/")
+    elif "STS2_MCP_BASE_URL" not in os.environ:
+        _base_url = f"http://{args.host}:{args.port}"

Troubleshooting

  • If Windows curl.exe http://127.0.0.1:15526/ fails, the game or mod is not ready.
  • If Windows curl.exe http://${sts2McpHost}:15527/ fails, the portproxy is not forwarding.
  • If Windows works but WSL fails, check the vEthernet (WSL) IP and firewall rules.
  • If WSL works but Codex fails, restart Codex so it reloads ~/.codex/config.toml.
  • If it breaks after reboot, refresh the adapter IP and update the --base-url.
  • If a fresh shell does not know $sts2McpHost, rerun the Get-NetIPAddress command or inspect netsh interface portproxy show v4tov4.

Safety Notes

  • The game mod starts on 127.0.0.1:15526, which is local to the host machine.
  • The portproxy is the part that makes the API reachable from WSL. Keep it bound to the WSL adapter IP, not 0.0.0.0.
  • The firewall rule should be narrow: the WSL adapter local address, port 15527, and the current WSL VM remote IP.
  • The API has no authentication token. Treat anything that can reach it as having full game control.
  • The mod is a plain local HTTP server. Browser pages and local processes can attempt localhost requests, so do not copy this pattern to more sensitive APIs without authentication and CSRF protections.
  • Do not expose this endpoint through Cloudflare Tunnel, ngrok, router port forwarding, or a public IP. Anyone who can reach the endpoint can potentially issue game actions.
